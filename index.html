Topic 2 — Tools and Commands

Extend Claude Code with custom slash commands and connect it to the outside world using the Model Context Protocol (MCP).

⌨️
Create Custom Commands
3 concepts ▼
How Commands Work
  • Create a .md file in .claude/commands/
  • The filename becomes the command name
  • The file content becomes Claude's instruction prompt
  • Trigger it with /command-name in Claude Code
  • Restart Claude to load newly created commands
$ARGUMENTS Placeholder
  • Place $ARGUMENTS anywhere in the .md
  • Everything typed after the command name is substituted in
  • Enables parameterised, reusable commands
# Example call with arguments
/write_tests src/lib/auth.ts
audit.md — No Arguments
Your goal is to update any vulnerable dependencies.

Do the following:
1. Run `npm audit` to find vulnerable packages
2. Run `npm audit fix` to apply updates
3. Run tests and verify nothing broke
Trigger: /audit
write_tests.md — With Arguments
Write comprehensive tests for: $ARGUMENTS

Testing conventions:
- Use Vitest with React Testing Library
- Place files in **tests** directory
- Name: [filename].test.ts(x)
- Test happy paths, edge cases, errors
Trigger: /write_tests <target>
Command Locations
  • .claude/commands/ — project-level, shared with team
  • ~/.claude/commands/ — personal global commands
  • Project commands take precedence over global ones
Best Practices
  • Write instructions as clear natural language tasks
  • Specify coding conventions and expected output format
  • Reference supporting files with @path/to/file
  • Use /help to list all commands available

Tools & MCP
4 concepts ▼
How Tool Use Works
  • Claude receives plain-text directions describing each tool
  • When Claude needs a tool it responds with a tool-use request
  • The Claude Code harness executes the tool call
  • Result is returned to Claude; it decides the next step
  • Claude understands what tools do & chooses intelligently
MCP — Model Context Protocol
  • Open standard connecting Claude to external services
  • Connect to: GitHub, Jira, Figma, Postgres, Slack, and more
  • Each server exposes Tools, Prompts, and Resources
  • Claude dynamically discovers available tools at startup
  • Tools using >10% of context window are deferred/lazy-loaded

Topic 3 — Skills and Agents

Extend Claude with reusable skill packages, run multiple parallel agents via git worktrees, automate GitHub workflows, and control tool behaviour with hooks.

🧠
Skills
3 loading levels ▼
What Are Agent Skills?
  • Modular, reusable capabilities for Claude
  • Package instructions, metadata & optional scripts
  • Unlike CLAUDE.md, skills are loaded on-demand
  • Idle skills cost zero context — only metadata is kept
  • Can be shared across projects and published to registries
SKILL.md vs CLAUDE.md
  • CLAUDE.md — always loaded, broad project context
  • SKILL.md — loaded only when triggered, task-specific
  • Skills compose: combine several for complex pipelines
  • Skills can spawn isolated sub-agents via context: fork
Three Loading Levels
1
Metadata — always loaded
YAML frontmatter (name, description). Claude knows the skill exists without loading its body. Zero context cost at idle.
2
Instructions — loaded when triggered
Main body of SKILL.md: workflows, best practices, step-by-step guidance. Loaded when a user request matches the skill's description.
3
Resources & Code — loaded as needed
Additional .md files, scripts, and templates bundled alongside SKILL.md. Pulled in only when referenced by the instructions.
SKILL.md Features
--- YAML frontmatter (Level 1 metadata) ---
name: security-review
description: Conduct a security review. Use when asked to audit vulnerabilities.
disable-model-invocation: true  # manual trigger only
context: fork             # spawns an isolated sub-agent
---

You are a senior security engineer. Conduct a thorough security review.

Multiple Agents & Sub Agents
2 patterns ▼
Parallel Agents via Git Worktrees
  • Each worktree = isolated file-system branch
  • Each runs its own independent claude session
  • Develop features in parallel without file conflicts
  • Main branch collects and merges all completed branches
# Create worktrees for 3 parallel features
mkdir .tree
git worktree add .tree/feature1
git worktree add .tree/feature2
git worktree add .tree/feature3

# Open a terminal in each, run claude
cd .tree/feature1 && claude

# On main: merge all features when done
# Then push to GitHub with /gitpush
Sub Agents
  • Specialised, autonomous AI instances within one session
  • Each gets its own fresh 200k-token context window
  • Only final results return to the main conversation
  • Up to 10 sub-agents can run simultaneously
  • Prevents main context from clutter of raw search data

Hooks
2 hook types ▼
Hooks let you run your own shell commands before or after any Claude Code tool call — optionally blocking the action entirely.
PreToolUse
Runs before the tool executes.

Your command can block the tool call — exit with a non-zero code and Claude receives your error message and tries an alternative.

Use cases: validate file paths, block restricted files, enforce naming conventions, deny deprecated function calls.
PostToolUse
Runs after the tool executes.

Too late to block — but your stdout is sent to Claude as additional feedback it can act on in its next step.

Use cases: run linter after edits, trigger tests after file changes, scan for TODO comments, audit logging.
Configuration (settings.json)
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read",          // intercept Read tool calls
        "hooks": [{
          "type": "command",
          "command": "node hooks/read_hook.js"
        }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",  // pipe-separate matchers
        "hooks": [{
          "type": "command",
          "command": "npx eslint --fix $CLAUDE_TOOL_INPUT_FILE_PATH"
        }]
      }
    ]
  }
}
Hook Locations
  • ~/.claude/settings.json — global, all projects
  • .claude/settings.json — project-level, committed
  • .claude/settings.local.json — project, not committed
Quick Setup
  • Use interactive /hooks command in Claude Code
  • Or write JSON directly in settings.json
  • Use the /update-config skill for guided setup
  • Hook receives tool metadata via environment variables
